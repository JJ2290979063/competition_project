# OS Agent — 操作系统智能代理

## 一、项目概述

### 1.1 项目背景

Linux 服务器是现代 IT 基础设施的核心，但传统的服务器管理方式存在诸多痛点：

- **命令行门槛高**：管理员需要记忆大量 Linux 命令及其参数组合，新手上手困难
- **交互成本大**：完成一个运维任务往往需要多条命令的组合执行，效率低下
- **误操作风险高**：一条错误的 `rm -rf` 或 `chmod` 命令就可能导致系统瘫痪，且缺乏有效的事前拦截机制
- **知识碎片化**：磁盘管理、进程管理、用户管理等知识分散在不同的工具和手册中

### 1.2 解决方案

OS Agent 是一个**自然语言驱动的 AI 智能代理**，用户只需用中文描述运维需求，Agent 即可自动理解意图、选择工具、执行命令并汇总结果。其核心特点包括：

- **自然语言交互**：用中文描述需求，无需记忆命令语法
- **智能决策**：基于 ReAct 范式的多步推理，自动拆解复杂任务
- **五级安全防护**：从 SAFE 到 CRITICAL 的分级风险控制，正则规则优先 + LLM 语义兜底的双层判断机制
- **多模式部署**：支持本地执行和 SSH 远程管理，提供 CLI 和 Web 两种交互界面
- **多轮对话记忆**：滑动窗口机制保留上下文，支持连续对话

### 1.3 技术栈

| 类别 | 技术选型 | 说明 |
|------|----------|------|
| 大语言模型 | Claude (Anthropic API) | 核心推理引擎，负责意图理解与决策 |
| Agent 框架 | 手动实现 ReAct 循环 | 基于 Anthropic SDK 的轻量级实现 |
| 工具定义 | LangChain Tools | 使用 `@tool` 装饰器定义标准化工具接口 |
| 远程执行 | Paramiko | SSH 协议远程命令执行，支持密钥和密码认证 |
| CLI 界面 | Rich + Click | 终端美化渲染、Markdown 输出、命令行参数解析 |
| Web 界面 | FastAPI + WebSocket | 实时双向通信，异步非阻塞架构 |
| 安全检测 | 正则引擎 + Claude API | 双层风险判断机制 |
| 配置管理 | python-dotenv | 环境变量驱动的配置管理 |

## 三、功能实现说明

### 3.1 基础能力

OS Agent 共实现 **15 个工具函数**，覆盖 Linux 服务器日常运维的四大核心场景：

#### 3.1.1 磁盘管理工具（`tools/disk.py`）

| 工具函数 | 功能描述 | 参数 | 底层命令 |
|----------|----------|------|----------|
| `check_disk_usage()` | 查看磁盘使用情况，返回各分区的挂载点、总容量、已用空间、可用空间和使用率。当某分区使用率超过 80% 时自动附加警告提示 | 无 | `df -h` |
| `check_disk_inode()` | 查看 inode 使用情况，返回各分区的 inode 总数、已用、可用和使用率 | 无 | `df -i` |
| `find_large_files(path, min_size_mb)` | 在指定路径下查找超过指定大小的大文件，返回前 20 条结果。内置路径安全校验，拒绝相对路径和路径遍历 | `path`: 搜索路径（默认 `/`）<br>`min_size_mb`: 最小文件大小 MB（默认 100） | `find ... -size +NM` |

#### 3.1.2 进程与端口管理工具（`tools/process.py`）

| 工具函数 | 功能描述 | 参数 | 底层命令 |
|----------|----------|------|----------|
| `list_processes(filter_name)` | 查看进程列表，支持按进程名过滤。未指定过滤条件时按 CPU 占用排序显示前 20 条 | `filter_name`: 进程名关键字（可选） | `ps aux` |
| `check_port(port)` | 检查指定端口的占用情况，返回监听地址、PID 和进程名。内置端口范围校验（1-65535） | `port`: 端口号 | `ss -tlnp` |
| `list_listening_ports()` | 查看所有正在监听的端口列表，包含端口号、监听地址和对应进程 | 无 | `ss -tlnp` |
| `kill_process(pid)` | 终止指定 PID 的进程。执行前先获取进程信息，拒绝操作 PID ≤ 1 的进程。此工具受安全模块审查 | `pid`: 进程 PID | `kill -9` |

#### 3.1.3 文件操作工具（`tools/file_ops.py`）

| 工具函数 | 功能描述 | 参数 | 底层命令 |
|----------|----------|------|----------|
| `list_directory(path)` | 查看目录内容，包含权限、大小、修改时间等详细信息。拒绝包含 `..` 的路径 | `path`: 目录路径 | `ls -lah` |
| `search_file(filename, search_path)` | 按文件名搜索文件，支持通配符，搜索深度限制 10 层，返回前 20 条结果 | `filename`: 文件名<br>`search_path`: 搜索路径（默认 `/`） | `find ... -name` |
| `show_file_content(path, lines)` | 查看文件内容，默认前 50 行，最多 200 行。内置敏感文件黑名单，拒绝读取 `/etc/shadow` 等文件 | `path`: 文件路径<br>`lines`: 显示行数（默认 50） | `head -n` |
| `get_file_info(path)` | 查看文件的详细元数据，包含大小、权限、所有者、时间戳等 | `path`: 文件路径 | `stat` |

#### 3.1.4 用户管理工具（`tools/user_mgmt.py`）

| 工具函数 | 功能描述 | 参数 | 底层命令 |
|----------|----------|------|----------|
| `list_users()` | 查看系统中可登录的正常用户列表，过滤 nologin 和 /bin/false 的系统账户，格式化输出用户名、UID、家目录和 Shell | 无 | 解析 `/etc/passwd` |
| `create_user(username, shell)` | 创建新系统用户，自动创建家目录。内置用户名正则校验（`^[a-zA-Z_][a-zA-Z0-9_]{2,31}$`）。此工具受安全模块审查 | `username`: 用户名<br>`shell`: Shell（默认 `/bin/bash`） | `useradd -m -s` |
| `delete_user(username, remove_home)` | 删除系统用户，可选是否同时删除家目录。硬编码禁止删除 root，拒绝删除 UID < 1000 的系统用户。此工具受安全模块审查 | `username`: 用户名<br>`remove_home`: 是否删除家目录（默认 `false`） | `userdel [-r]` |
| `check_user_exists(username)` | 检查指定用户是否存在，返回用户基本信息（UID、GID、所属组） | `username`: 用户名 | `id` |

### 3.2 安全风险控制（重点）

安全风险控制是本项目的核心亮点，采用**"正则规则优先 + LLM 语义兜底"的双层判断机制**，实现了从命令拦截到用户确认的完整安全闭环。

#### 3.2.1 五级风险体系

系统定义了五个风险等级，每个等级对应不同的处理策略：

| 风险等级 | 数值 | 处理策略 | 适用场景 | 用户交互 |
|----------|------|----------|----------|----------|
| `SAFE` | 0 | 直接执行 | 只读查询操作，不会对系统产生任何修改 | 无需确认，静默执行 |
| `LOW` | 1 | 执行并记录日志 | 轻微修改操作，影响范围有限 | 无需确认，记录审计日志 |
| `MEDIUM` | 2 | 提示风险后执行 | 中等影响操作，如安装软件包、创建用户 | 黄色面板提示，需输入 `yes` 确认 |
| `HIGH` | 3 | 必须二次确认 | 高风险操作，如删除用户、强制终止进程 | 橙色警告面板，需输入 `yes` 确认 |
| `CRITICAL` | 4 | 直接拦截拒绝 | 极危险操作，可能导致系统不可用 | 红色拦截面板，无法通过确认绕过 |

#### 3.2.2 正则规则详细说明

**CRITICAL 级别（12 条规则）— 直接拦截，不可绕过**

| 序号 | 拦截模式 | 风险说明 |
|------|----------|----------|
| 1 | `rm -rf /` 及其变体 | 删除根目录下所有文件，将摧毁整个系统 |
| 2 | `rm -rf /*` | 删除根目录下所有文件，将摧毁整个系统 |
| 3 | `rm ... /(etc\|boot\|bin\|sbin\|lib\|lib64\|usr)` | 删除系统核心目录，将导致系统无法启动 |
| 4 | `mkfs.*` | 格式化磁盘分区，将清除所有数据 |
| 5 | `dd if=/dev/zero of=/dev/sdX` | 直接向磁盘写零/随机数据，将摧毁磁盘数据 |
| 6 | `> /dev/sdX` | 直接重定向到磁盘设备，将破坏分区表 |
| 7 | `userdel.*root` | 删除 root 用户将导致系统完全不可用 |
| 8 | `chmod 000 /` | 将根目录权限设为 000，系统将完全无法访问 |
| 9 | `chmod 000 /(etc\|bin\|sbin\|lib\|usr)` | 将系统核心目录权限设为 000，系统将无法运行 |
| 10 | `chown -R ... /` | 递归更改根目录所有者，将破坏系统权限 |
| 11 | `:(){ :\|:& };:` (fork bomb) | 耗尽系统资源导致系统崩溃 |
| 12 | `> /etc/(passwd\|shadow\|group\|sudoers)` | 直接覆盖关键系统文件 |

**HIGH 级别（10 条规则）— 需要用户明确确认**

| 序号 | 检测模式 | 风险说明 |
|------|----------|----------|
| 1 | `userdel` | 删除系统用户是不可逆操作 |
| 2 | 编辑 `/etc/sudoers`、`/etc/passwd`、`/etc/shadow`、`/etc/fstab`、`/etc/ssh` | 修改关键系统配置文件 |
| 3 | `systemctl stop/disable/mask` 关键服务 | 停止 sshd、firewalld、iptables 等可能导致远程连接中断 |
| 4 | `kill -9` / `kill -KILL` | 强制终止进程可能导致数据丢失 |
| 5 | `iptables -A/-D/-I/-F` | 修改防火墙规则可能导致网络连接中断 |
| 6 | `firewall-cmd` | 修改防火墙规则可能导致网络连接中断 |
| 7 | `chmod -R` 对 `/home`、`/var`、`/opt`、`/srv` | 递归权限变更影响范围大 |
| 8 | `rm -rf` (任意递归删除) | 递归删除文件，需确认目标路径正确 |
| 9 | `shutdown`、`reboot`、`poweroff`、`init [0-6]` | 系统关机/重启操作 |
| 10 | `passwd`、`crontab -r` | 修改密码或删除所有定时任务 |

**MEDIUM 级别（8 条规则）— 提示风险，用户知情后执行**

| 序号 | 检测模式 | 风险说明 |
|------|----------|----------|
| 1 | `useradd` | 创建新用户会增加系统访问入口 |
| 2 | `yum/apt/dnf/pip install/remove` | 安装/卸载软件包会改变系统环境 |
| 3 | `systemctl restart/start/enable` | 重启/启动服务可能影响正在运行的业务 |
| 4 | 编辑 `/etc/` 下的文件 | 修改系统配置文件 |
| 5 | `sed -i` | sed 就地修改文件 |
| 6 | `chmod` | 修改文件权限 |
| 7 | `chown` | 修改文件所有者 |
| 8 | `mv /etc/`、`cp ... /etc/` | 移动或覆盖系统配置文件 |

**SAFE 级别（8 条规则）— 只读操作，直接放行**

| 序号 | 匹配模式 | 覆盖命令 |
|------|----------|----------|
| 1 | 系统信息查询 | `df`、`du`、`free`、`uptime`、`uname`、`hostname`、`whoami`、`id`、`w`、`who`、`last`、`date`、`cal` |
| 2 | 进程/网络查看 | `ps`、`top`、`htop`、`ss`、`netstat`、`lsof`、`ip`、`ifconfig` |
| 3 | 文件读取 | `ls`、`ll`、`cat`、`head`、`tail`、`less`、`more`、`wc`、`file`、`stat`、`type`、`which`、`whereis` |
| 4 | 搜索命令 | `find`、`locate`、`grep`、`egrep`、`fgrep`、`awk`、`sort`、`uniq`、`cut`、`tr` |
| 5 | sed 只读模式 | `sed`（不带 `-i` 参数） |
| 6 | 输出命令 | `echo`、`printf`、`test`、`[` |
| 7 | 服务状态查询 | `systemctl status/is-active/list` |
| 8 | 日志与包查询 | `journalctl`、`dmesg`、`rpm -q`、`dpkg -l`、`pip list`、`pip show` |

#### 3.2.3 双层判断机制

```
命令输入
    │
    ▼
┌─────────────────────────────────┐
│  第一层：正则规则快速匹配         │
│                                 │
│  1. 检查 SAFE 规则 → 匹配则放行   │
│  2. 检查 CRITICAL 规则 → 匹配则拦截│
│  3. 检查 HIGH 规则 → 匹配则确认   │
│  4. 检查 MEDIUM 规则 → 匹配则提示  │
│  5. 均未匹配 → 进入第二层          │
└────────────┬────────────────────┘
             │ 规则无法判断
             ▼
┌─────────────────────────────────┐
│  第二层：Claude API 语义分析      │
│                                 │
│  输入：命令文本 + 执行上下文       │
│  评估维度：                       │
│    ① 是否影响系统核心文件/目录     │
│    ② 是否不可逆（删除、格式化等）  │
│    ③ 影响范围（单文件/批量/系统级） │
│    ④ 是否涉及权限或安全配置        │
│  输出：结构化 JSON 风险结果        │
│                                 │
│  异常兜底：LLM 调用失败时          │
│  默认返回 LOW 级别                │
└─────────────────────────────────┘
```

这种设计的优势在于：

1. **性能优先**：已知危险模式通过正则规则毫秒级匹配，无需等待 API 调用
2. **覆盖全面**：正则规则覆盖 38 条已知模式，LLM 兜底处理未知命令组合
3. **安全兜底**：即使 LLM 调用失败，也不会将未知命令判定为安全，而是默认 LOW 级别并记录日志
4. **可解释性**：每个风险结果都包含中文原因说明和操作建议

#### 3.2.4 二次确认交互流程

当命令被判定为 MEDIUM 或 HIGH 级别时，系统会通过 Rich 库渲染彩色警告面板：

```
┌──────────────────────────────────────────┐
│  ⚠ 高风险操作警告                         │
│                                          │
│  风险等级：HIGH                           │
│  风险原因：高风险操作：删除系统用户是不可逆操作│
│  执行命令：userdel testuser               │
│  建议：需要您的明确确认后才能执行            │
│                                          │
│  是否继续执行？请输入 yes / no：            │
└──────────────────────────────────────────┘
```

- 用户输入 `yes`：执行操作并记录审计日志
- 用户输入 `no`：取消操作，Agent 收到取消通知后调整后续行为
- CRITICAL 级别：直接显示红色拦截面板，不提供确认选项

### 3.3 多轮对话能力

#### 3.3.1 滑动窗口记忆机制

OS Agent 采用基于 `collections.deque` 的滑动窗口记忆机制，保留最近 **10 轮**对话（即 20 条消息：10 条 `HumanMessage` + 10 条 `AIMessage`）。

```python
class AgentMemory:
    def __init__(self, window_size: int = 10):
        self._messages: deque = deque(maxlen=window_size * 2)
```

设计要点：

- **固定窗口大小**：使用 `deque(maxlen=20)` 自动淘汰最早的消息，无需手动管理
- **双消息结构**：每轮对话由 `HumanMessage` 和 `AIMessage` 成对存储，保证上下文完整性
- **完整日志分离**：`_full_log` 列表记录所有对话（含时间戳），用于导出和审计，不受窗口限制
- **格式化输出**：`get_history()` 方法返回 `[用户]` / `[助手]` 前缀的格式化文本，便于展示

#### 3.3.2 上下文构建流程

每次用户输入时，Agent 会：

1. 从 `AgentMemory` 获取历史消息列表
2. 将历史消息转换为 Anthropic API 的 `messages` 格式（`user` / `assistant` 角色）
3. 追加当前用户输入
4. 将完整消息列表发送给 Claude API
5. 对话结束后，将本轮交互保存到记忆中

这确保了 Agent 在多轮对话中能够理解上下文引用（如"刚才那个进程"、"再查一下那个目录"等）。

### 3.4 Agent 决策机制

#### 3.4.1 ReAct 手动循环

OS Agent 采用手动实现的 **ReAct（Reasoning + Acting）** 循环，而非依赖 LangChain 的 AgentExecutor。这种设计提供了更精细的控制能力。

```
┌─────────────────────────────────────────────────┐
│              ReAct 循环（最多 5 次）               │
│                                                 │
│  ┌──────────┐                                   │
│  │ 调用 LLM  │ ← 系统提示 + 历史消息 + 用户输入   │
│  └────┬─────┘                                   │
│       │                                         │
│       ▼                                         │
│  ┌──────────────┐                               │
│  │ 解析 LLM 输出 │                               │
│  └────┬─────────┘                               │
│       │                                         │
│       ├── 无 JSON → 视为自然语言回答 → 返回       │
│       │                                         │
│       ├── action = "final_answer" → 提取答案 → 返回│
│       │                                         │
│       ├── action = 未知工具 → 追加错误信息 → 继续  │
│       │                                         │
│       └── action = 已知工具                      │
│               │                                 │
│               ▼                                 │
│         ┌──────────────┐                        │
│         │ 安全审查       │                        │
│         │ (仅限高危工具)  │                        │
│         └────┬─────────┘                        │
│              │                                  │
│              ├── 拦截 → 追加拦截信息 → 继续        │
│              │                                  │
│              └── 放行 → 执行工具                  │
│                      │                          │
│                      ▼                          │
│                ┌──────────┐                     │
│                │ 工具执行   │                     │
│                └────┬─────┘                     │
│                     │                           │
│                     ▼                           │
│              追加执行结果到消息列表                │
│                     │                           │
│                     └──→ 回到循环顶部             │
│                                                 │
│  循环结束后：最终调用一次 LLM 生成总结性回答        │
└─────────────────────────────────────────────────┘
```

#### 3.4.2 工具选择逻辑

工具选择完全由 LLM 驱动，通过 System Prompt 中的工具目录描述，Claude 根据用户意图自主选择合适的工具：

- LLM 输出严格的 JSON 格式：`{"action": "tool_name", "action_input": {"param": "value"}}`
- 解析器 `_parse_llm_response()` 使用正则提取 JSON，兼容 Markdown 代码块包裹
- 工具注册表 `TOOL_REGISTRY` 维护 15 个工具函数的映射
- 仅 `kill_process`、`create_user`、`delete_user` 三个高危工具需要经过安全审查

#### 3.4.3 循环控制

- **最大迭代次数**：`MAX_ITERATIONS = 5`，防止无限循环
- **提前退出**：当 LLM 返回 `final_answer` 或纯文本时立即退出
- **错误恢复**：未知工具名、安全拦截等情况下，将错误信息反馈给 LLM，由其调整策略
- **兜底机制**：循环耗尽后，强制调用一次 LLM 生成总结性回答

## 四、核心 Prompt 设计

### 4.1 系统提示词（SYSTEM_PROMPT）

以下是 `agent/prompts.py` 中定义的完整系统提示词：

```
你是一个专业的 Linux 服务器管理助手，负责帮助用户通过自然语言完成服务器管理任务。

你的工作原则：
- 准确理解用户意图，选择合适的工具执行操作
- 优先使用只读命令获取信息，再考虑修改类命令
- 对任何可能影响系统稳定性的操作，主动说明风险
- 操作完成后，用清晰的中文总结执行结果
- 如果用户的指令不够明确，先询问澄清再执行
- 如果一个任务需要多步操作，逐步执行并报告每一步的结果

你可以使用以下工具：
- check_disk_usage: 查看磁盘使用情况，无需参数
- check_disk_inode: 查看 inode 使用情况，无需参数

- list_processes: 查看进程列表，参数：filter_name(str，可选)

- kill_process: 结束进程，参数：pid(int)
- list_directory: 查看目录内容，参数：path(str)
- search_file: 搜索文件，参数：filename(str), search_path(str)
- show_file_content: 查看文件内容，参数：path(str), lines(int)
- get_file_info: 查看文件信息，参数：path(str)
- list_users: 查看用户列表，无需参数
- create_user: 创建用户，参数：username(str)

当你需要调用工具时，必须严格按照以下 JSON 格式输出，不要输出其他内容：
{"action": "工具函数名", "action_input": {"参数名": "参数值"}}

当你已经得到工具执行结果并可以回答用户时，输出：
{"action": "final_answer", "action_input": "你的最终回复内容"}

安全限制：
- 禁止执行任何可能损坏系统的命令
- 对高风险操作必须获得用户明确确认
- 永远不会执行被安全系统标记为 CRITICAL 的命令
- 读取文件时拒绝访问 /etc/shadow 等敏感文件

输出要求：
- 所有面向用户的回复使用中文
- 技术细节（如命令、路径）可保留英文
- 表格类数据尽量格式化展示
- 必须严格按照上述 JSON 格式输出，不要在 JSON 外添加任何文字
```

### 4.2 Prompt 设计思路

#### 4.2.1 角色定位

Prompt 开篇即明确 Agent 的角色——"专业的 Linux 服务器管理助手"，这为后续所有行为设定了边界：只处理服务器管理相关任务，不涉及其他领域。

#### 4.2.2 六条工作原则

| 原则 | 设计意图 |
|------|----------|
| 准确理解用户意图，选择合适的工具 | 引导 LLM 进行意图分析而非直接执行 |
| 优先使用只读命令 | **安全优先**：先查看再修改，降低误操作风险 |
| 主动说明风险 | 让用户知情，配合安全模块的二次确认机制 |
| 用清晰的中文总结结果 | 降低用户理解门槛，无需解读原始命令输出 |
| 指令不明确时先询问 | 避免歧义导致的错误操作 |
| 多步任务逐步执行 | 利用 ReAct 循环的多轮能力，每步可审查 |

#### 4.2.3 工具目录设计

Prompt 中列出了全部 15 个工具的名称、功能描述和参数规格。这种"工具目录"式的设计让 LLM 能够：

- 根据用户意图精确匹配工具
- 正确构造工具调用参数
- 在多个候选工具中做出合理选择

#### 4.2.4 输出格式约束

严格要求 JSON 格式输出是 ReAct 循环正常运转的关键：

- **工具调用**：`{"action": "tool_name", "action_input": {...}}`
- **最终回答**：`{"action": "final_answer", "action_input": "回答文本"}`
- **格式兜底**：解析器 `_parse_llm_response()` 兼容 Markdown 代码块包裹的 JSON

#### 4.2.5 安全约束嵌入

在 Prompt 层面嵌入安全约束，形成与代码层安全检查的**双重保障**：

- Prompt 层：引导 LLM 自觉避免危险操作
- 代码层：`RiskDetector` 在工具执行前强制拦截

### 4.3 风险分析提示词（RISK_ANALYSIS_PROMPT）

当正则规则无法判断命令风险时，系统使用以下提示词调用 Claude API 进行语义分析：

```
分析以下 Linux 命令的安全风险等级。

命令：{command}
执行上下文：{context}

请从以下维度评估：
1. 是否影响系统核心文件或目录
2. 是否不可逆（删除、格式化等）
3. 影响范围（单个文件 vs 批量 vs 系统级）
4. 是否涉及权限或安全配置

返回严格的 JSON 格式（不要包含任何其他内容）：
{
  "level": "SAFE|LOW|MEDIUM|HIGH|CRITICAL",
  "reason": "风险原因（中文）",
  "suggestion": "建议（中文）",
  "blocked": true或false
}
```

该提示词的设计要点：

- **四维度评估框架**：从系统文件影响、可逆性、影响范围、权限安全四个维度全面评估
- **结构化输出**：要求返回严格 JSON，便于程序解析
- **上下文感知**：传入执行上下文（如工具名称），帮助 LLM 理解命令的使用场景

## 七、部署说明

### 7.1 环境要求

| 项目 | 要求 |
|------|------|
| 操作系统 | Linux / macOS / Windows（推荐 Linux） |
| Python | >= 3.10 |
| 网络 | 需要访问 Anthropic API（或兼容的代理端点） |
| SSH（远程模式） | 目标服务器需开放 SSH 端口，且有合法的登录凭据 |

### 7.2 安装步骤

```bash
# 1. 克隆项目
git clone <repository-url>
cd os-agent

# 2. 创建虚拟环境（推荐）
python -m venv venv
source venv/bin/activate  # Linux/macOS
# 或 venv\Scripts\activate  # Windows

# 3. 安装依赖
pip install -r requirements.txt

# 4. 配置环境变量
cp .env.example .env
# 编辑 .env 文件，填入实际配置
```

### 7.3 启动命令

#### CLI 模式（默认）

```bash
# 本地模式
python main.py

# SSH 远程模式
python main.py --mode remote --host 192.168.1.100 --user root

# 指定 SSH 端口
python main.py --mode remote --host 192.168.1.100 --port 2222 --user admin

# 调试模式（控制台输出详细日志）
python main.py --debug
```

#### Web 模式

```bash
# 启动 Web 界面（默认监听 0.0.0.0:8000）
python main.py --web

# Web + 远程模式
python main.py --web --mode remote --host 192.168.1.100
```

启动后在浏览器访问 `http://localhost:8000` 即可使用 Web 界面。

### 7.4 配置项说明

| 配置项 | 必填 | 默认值 | 说明 |
|--------|------|--------|------|
| `ANTHROPIC_API_KEY` | 是 | — | Anthropic API 密钥，用于调用 Claude 模型 |
| `ANTHROPIC_BASE_URL` | 否 | `https://ruoli.dev` | API 端点地址，支持自定义代理 |
| `SSH_HOST` | 远程模式必填 | `localhost` | SSH 目标主机地址 |
| `SSH_PORT` | 否 | `22` | SSH 端口号 |
| `SSH_USER` | 远程模式必填 | `root` | SSH 登录用户名 |
| `SSH_PASSWORD` | 否 | — | SSH 密码（与密钥二选一） |
| `SSH_KEY_PATH` | 否 | — | SSH 私钥路径（优先于密码认证） |
| `AGENT_MODE` | 否 | `local` | 运行模式：`local`（本地）或 `remote`（SSH 远程） |
| `REQUIRE_CONFIRM_FOR_HIGH_RISK` | 否 | `true` | 是否对高风险操作启用二次确认 |
| `MAX_RETRY_COUNT` | 否 | `3` | 最大重试次数 |

### 7.5 命令行参数

| 参数 | 类型 | 说明 |
|------|------|------|
| `--mode` | `local` \| `remote` | 运行模式，覆盖 `.env` 中的 `AGENT_MODE` |
| `--host` | string | SSH 主机地址，覆盖 `.env` 中的 `SSH_HOST` |
| `--port` | int | SSH 端口，覆盖 `.env` 中的 `SSH_PORT` |
| `--user` | string | SSH 用户名，覆盖 `.env` 中的 `SSH_USER` |
| `--web` | flag | 启动 Web 界面（默认启动 CLI） |
| `--debug` | flag | 启用调试模式，控制台输出详细日志 |
